05 — Data Flows

Control & Data Flow

Three primary flows: Remember (save), Ask (retrieve), and Settings Change (reindex).

Remember Mode — Save a Memory
1
User submits in Composer
Text / image blob / voice recording. Mode = "remember" detected by trigger words.
2
Save raw asset
Image or audio blob persisted to assets table with auto-delete schedule.
3
Multimodal extraction
Audio → Whisper/OS Speech → text. Image → Vision API → text. Text normalized.
4
Memory extraction (AI)
AI classifies memory_type, extracts canonical_text, subject, confidence.
5
Embed & duplicate check
Generate 256/1536-dim embedding. Cosine similarity against all active memories. Warn if >0.82.
6
Persist to IndexedDB
memory_items + memory_embeddings written atomically. Previous conflicting memory marked superseded.
7
Confirmation reply
Assistant message with success/duplicate/conflict status. UI refreshes.
Ask Mode — RAG Retrieval Pipeline
1
User submits question
Text or voice. Voice transcribed first. Mode = "ask" (no trigger words).
2
Embed query
Same embedding provider as memories. Ensures vector space alignment.
3
Cosine similarity scan
Dot product against all memory_embeddings. Filter >0.02 threshold.
4
Rerank candidates
Composite score = semantic + age_boost + confidence×0.15 + subject_match + type_match - superseded_penalty.
score = semantic + age_boost // recent = higher + confidence × 0.15 // memory quality + 0.5 (exact subj) // or 0.25 (contains) + 0.15 (type match) // inferred type - 0.25 (superseded) // penalize stale
5
Select top-3
Top candidates formatted with date, source_kind, confidence into reasoning context.
6
AI reasoning
ReasoningProvider answers using only provided context. States uncertainty explicitly.
7
Format & display answer
Response appended to chat history. Low-confidence memories flagged inline.
Settings Change + Auto Re-index Flow
sequenceDiagram participant U as User participant SP as SettingsPanel participant UMC as useMemoryCompanion participant DB as IndexedDB participant AI as AI Registry U->>SP: Change API key or provider SP->>UMC: updateSettings(nextSettings) UMC->>DB: saveSettings(nextSettings) UMC->>UMC: Detect key/provider change alt key or provider changed UMC->>UMC: setReindexing(true) UMC->>AI: getAIRegistry(nextSettings) loop For each active memory AI->>AI: embeddingProvider.embed(canonicalText) AI->>DB: Update memory_embeddings end UMC->>UMC: setReindexing(false) end UMC->>U: UI shows "Indexing…" → done

07 — Voice Input

Dual Voice Input Architecture

Two completely different paths depending on whether an OpenAI key is available.

☁️ Cloud Path (OpenAI key present)

  • Hook: useVoiceRecorder
  • MediaRecorder API (webm/opus/ogg/mp4)
  • Real-time audio level via AnalyserNode
  • Stop → Blob → FormData
  • POST to OpenAI /audio/transcriptions
  • Model: gpt-4o-mini-transcribe
  • Returns transcribed text → memory pipeline

🎤 OS Path (Anthropic-only)

  • Hook: useSpeechRecognition
  • Web Speech API (SpeechRecognition)
  • Continuous mode + interim results
  • iOS: Safari native speech engine
  • Android: Google Speech Services
  • Desktop: OS speech recognition
  • Live transcript → memory pipeline

10 — Security

Security Model

Privacy-by-design: no server, no analytics, no tracking. API keys never leave the device except in direct calls to provider endpoints.

Security Controls
✓
API keys in IndexedDB only
Never in env vars, config files, or the JS bundle. Entered at runtime via Settings UI.
✓
Direct-to-provider API calls
Fetch calls go browser → OpenAI/Anthropic. Zero proxy, zero server log.
✓
PBKDF2 passcode hashing
Web Crypto API hashes passcode before storage. Constant-time verification. No plaintext.
✓
Raw media auto-deletion
Image and audio blobs purged after configurable retention period (default 7 days). Transcript text retained.
✓
Storage persistence API
navigator.storage.persist() requested on first launch to protect against browser eviction.
!
Data stored plaintext in IndexedDB
Passcode locks the UI but does not encrypt the DB. Physical device access = data access.
Threat Model Summary
ThreatStatus
Network interception of keysMitigated — HTTPS + direct calls
Server-side data breachN/A — no server
Malicious JS in bundleMitigated — open source, Vite build
Browser eviction of DBMitigated — Storage persist API
Shoulder surfingMitigated — Passcode lock
Physical device accessPartial — UI locked, DB plaintext
iOS PWA deletionUser risk — wipes all data
API key leakage via logsMitigated — no server logs
11 — Algorithms

Key Algorithms

Embedding, similarity, and reranking logic that powers memory retrieval.

Local Embedding (n-gram hashing)
// 256-dimensional sparse vector function embed(text) { tokens = tokenize(text) // lowercase, split stopwords = ["the", "a", "is", "in"...] for token in tokens: hash = fnv32(token) % 256 vec[hash] += 2 // token weight for trigram in ngrams(tokens, 3): hash = fnv32(trigram) % 256 vec[hash] += 1 // 3-gram weight for fourgram in ngrams(tokens, 4): hash = fnv32(fourgram) % 256 vec[hash] += 1 // 4-gram weight return l2_normalize(vec) // unit vector }
Retrieval Reranking
// Composite reranking score function rerank(candidate, query) { score = cosine(query_vec, mem_vec) // semantic age = daysSince(candidate.created_at) score += max(0, (30 - age) / 30 * 0.2) score += candidate.confidence * 0.15 if (subjectMatch == "exact"): score += 0.5 elif (subjectMatch == "contains"): score += 0.25 if (inferredType == candidate.memory_type): score += 0.15 if (candidate.superseded_by): score -= 0.25 return score } // Select top-3 by final score // Minimum semantic threshold: 0.02
